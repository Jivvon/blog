ref. About the Open Container Initiative : https://opencontainers.org/about/overview/

> The OCI currently contains three specifications: the Runtime Specification (runtime-spec), the Image Specification (image-spec) and the Distribution Specification (distribution-spec)
> OCI(Open Container Initiative)는 컨테이너 기술의 표준을 정의하는 프로젝트이고, 여기서 정하는 핵심 표준은 크게 3가지이다.

1. Runtime Spec(런타임 표준)
2. Image Spec(이미지 표준)
3. Distribution Spec(배포 표준)

### OCI Runtime Spec

> The Runtime Specification outlines how to run a “filesystem bundle” that is unpacked on disk.
> At a high-level an OCI implementation would download an OCI Image then unpack that image into an OCI Runtime filesystem bundle.
> At this point the OCI Runtime Bundle would be run by an OCI Runtime.

high-level OCI 구현은 OCI 이미지를 다운로드 한 후 해당 이미지를 OCI 런타임 파일 시스템 번들로 압축 해제할 때, 이 번들이 OCI 런타임에 의해 실행된다.

이 전체 워크플로우는 컨테이너 엔진에서 추가적인 Argument 없이 이미지를 실행할 수 있는 UX를 지원해야 한다.

### OCI Image Spec

> To support this UX the OCI Image Format contains sufficient information to launch the application on the target platform (e.g. command, arguments, environment variables, etc).
> This specification defines how to create an OCI Image, which will generally be done by a build system, and output an image manifest, a filesystem (layer) serialization, and an image configuration.

OCI Image Format에는 대상 플랫폼에서 어플리케이션을 시작하는 데 필요한 정보가 포함되어 있다
OCI Image Spec에는 일반적으로 빌드 시스템에서 수행되는 OCI 이미지를 만드는 방법과 _an image manifest, a filesystem (layer) serialization, and an image configuration_ 을 정의한다.

OCI Image Spec에 대해서 자세히 알아보자

ref. https://github.com/opencontainers/image-spec

**OCI Image Spec**은 컨테이너 이미지의 building, transporting, preparing(실행 전 단계)까지 표준화를 목표로 정의된 스펙이며,
Manifest, Index, Layers, Config로 구성되어 있다.

- **Image Manifest**: 이미지 구성 요소(파일시스템 레이어, 의존성 등) 메타데이터를 담고 있음.
- **Image Index**: 여러 플랫폼(아키텍처)에 따른 다양한 Manifest들의 집합.
- **Filesystem Layers**: 컨테이너 파일시스템의 아카이브 모음.
- **Configuration**: 이미지 레이어 순서, 환경 변수, 명령어 등 실행에 필요한 구성 정보.

먼저, 어떤 미디어 타입이 있는지 훑어보는 것이 OCI 이미지 명세의 전체 구조를 이해하는 데 도움이 된다.

미디어 타입은 다음과 같은 규칙을 따른다.

```text
application/vnd.oci.\<component\>.\<version\>+\<format\>
```

- **application/vnd.oci**: application은 MIME 타입의 최상위 타입, vnd는 vendor
- **\<component\>**: 데이터 종류 (image.manifest, image.config, image.layer, image.index, layout.header)
- **\<version\>**: 버전. 현재는 v1을 사용
- **\<format\>**: 데이터가 실제로 저장된 파일 포맷. 대부분 json이고, 파일시스템 아카이브 모음인 Layer는 tar archive와 압축 방법을 함게 표기함

```bash
- application/vnd.oci.image.manifest.v1+json	# 이미지 매니페스트 (JSON 포맷)
- application/vnd.oci.image.index.v1+json	    # 매니페스트의 인덱스, 멀티 아키텍처 지원 (JSON)
- application/vnd.oci.image.layer.v1.tar    	# 이미지의 실제 데이터 레이어 (tar 포맷)
- application/vnd.oci.image.layer.v1.tar+gzip	# gzip으로 압축된 레이어 데이터
- application/vnd.oci.layout.header.v1+json	    # 레이아웃 헤더 정의 (JSON)
```

**image manifest** [#](https://github.com/opencontainers/image-spec/blob/main/manifest.md)

- media type: `application/vnd.oci.image.manifest.v1+json`

image manifest의 목적 중 하나는 **멀티 아키텍처 지원**이다. kubernetes에서 arm64, amd64 아키텍처 머신에 모두 스케줄링 될 수 있는 경우, 각 아키텍처에 맞는 이미지가 모두 빌드되어 있어야 한다.
Pod이 스케줄링되어 컨테이너를 생성할 때, 컨테이너 런타임은 manifest를 보고 아키텍처에 맞는 이미지의 digest를 내려받는다.
빌드할 때에는 `docker buildx build --platform=linux/arm64,linux/amd64` 처럼 여러 플랫폼을 한 번에 빌드할 수 있다.

**image configuration**

> The image configuration includes information such as application arguments, environments, etc. The combination of the image manifest, image configuration, and one or more filesystem serializations is called the OCI Image.

image configuration에는 argouemtns, environments 등의 정보가 포함된다.
image manifest, image configuration및 하나 이상의 파일 시스템 직렬화를 결합한 것을 OCI 이미지라고 합니다.
